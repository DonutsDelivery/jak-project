# Minimal Target Subsystem Port Plan — Jak X Bringup

Research-only. Maps the shortest path from the current state (title menu renders; START press → `initialize! 'game` → spawned lambda → stub `start` returning `#f`) to a state where `start` spawns a real `target` process. Goal is *not* a playable race — just a live target that survives the boot lambda and lets `*target*` stop being `#f`.

## Key correction to the brief

**Jak X DOES have a `target` type.** The comment in `goal_src/jakx/engine/util/jakx-stubs.gc:188` ("Jak X is a racing game and has no `target`") is wrong and was speculation. Evidence:

- `decompiler_out/jakx/target-h_disasm.gc` registers `target` as `process-focusable` subtype, size `0x3c02c00350` (same encoding scheme as jak3).
- `decompiler_out/jakx/logic-target_ir2.asm` contains real bodies for `start`, `stop`, `start-1`, `start-2`, `target-init`, `(method 10 target)`, `(method 59 target)` (= `target-init!`), `reset-target-state`, `target-post`, `target-no-move-post`, `target-real-post`, `level-setup`, `target-move-dist`, `target-print-stats`, `target-history-print`, `draw-history`.
- `decompiler_out/jakx/target-death_ir2.asm` contains `target-continue` state + `target-continue-init-art`, `arrive-at-continue`, `next-continue` + 6 anonymous state-handler lambdas.
- `decompiler_out/jakx/title-obs_ir2.asm:244` registers `target-title` state with code `sleep-code` and post `post-memcard` — **the title-screen target state lives in title-obs, not target2, and is trivial (sleep).**

So the jakx target is jak-the-character wearing a vehicle *as gameplay*. The core target.gc code treats him as a process-focusable actor. Vehicle-specific logic (`vehicle-h`, `vehicle`, `car-info-h`, `net-race`, `race-*`) is orthogonal — `target` owns the player, and a vehicle is a separate process the target rides via a `pilot-info`-like mechanism. This research only covers the target subsystem.

## 1. What `start` actually has to do

The shortest viable `start` (jak3 `logic-target.gc:3155`; jakx MIPS at `logic-target_ir2.asm:310`) is 15 lines of GOAL:

```gc
(defun start ((arg0 symbol) (arg1 continue-point))
  (set! (-> *level* play?) (if (= arg0 'play) #t (-> *setting-control* user-default border-mode)))
  (set! (-> *setting-control* user-default border-mode) #f)
  (set! (-> *setting-control* user-default region-mode) #f)
  (apply-settings *setting-control*)
  (stop arg0)
  (let ((v1 (process-spawn target :init target-init arg1
                           :name "target"
                           :from *target-dead-pool*
                           :to *target-pool*
                           :stack *kernel-dram-stack*)))
    (if v1
        (set! *target* (the-as target (-> v1 0 self)))
        (set! *target* #f)))
  *target*)
```

Our current stub already does the settings/stop prelude. The missing piece is the `process-spawn target :init target-init` call. Doing that unlocks:

- `*target*` becomes non-#f for downstream `when *target*` checks.
- `target-init` (jak3:3099 / jakx `logic-target_ir2.asm:943`, ~60 lines MIPS) runs as the init hook inside the spawn.
- `target-init` calls `target-init!` (method 59; jakx has body at 491 MIPS lines, jak3 is the 260-line `defmethod` at `logic-target.gc:2831`) then `(go target-continue arg0)`.

After spawn, the target process is parked in `target-continue` state — which is a pure settings/fade-out state, not gameplay.

## 5. Minimum viable path — one-session feasibility

**The minimum path to "`*target*` is non-#f after initialize! 'boot" is:**

1. Real `start` (replace current stub — ~15 lines).
2. Real `stop` (already done).
3. `target-init` behavior (~10 lines).
4. `target-init!` method — the hard one. ~100 GOAL lines. Needs careful offset tracking against target-h, plus:
   - `collide-shape-moving` + `collide-shape-prim-*` must be new-able from `'process` allocation. Verify jakx has these deftypes and they link.
   - `view-get-active-index` must exist.
   - `*standard-dynamics*` must exist.
   - `cshape-reaction-default` (or a substitute) must exist.
5. `reset-target-state` (~20 lines) — needs `target-exit` (~15 lines) and `buzz-stop!`.
6. `target-generic-event-handler` — **stub to `(lambda () 0)`** category E. 2 lines.
7. `target-continue` state — **stub to minimum `:code sleep-code :post target-no-move-post`** category E. 5 lines.
8. `target-no-move-post` + `target-post` + `target-real-post` (~10 lines total, mechanical).
9. `level-setup` (~30 lines).
10. `*target-dead-pool*` + `*target-pool*` globals — verify allocated in kernel init, else add.
11. Shrink `target-h.gc`'s extern list so linker doesn't fail on undefined symbols.

## 6. Does `control-spec` state matter?

No. `control-spec` in jakx `title-obs.gc:230` is a virtual state on `title-control` for the "controller customization" sub-menu. It's a sleep-code stub in our port. The brief's mention of "(start 'play ...)` calls" in `control-spec` is wrong — jakx's control-spec state does not call `start`. The `(start 'play ...)` macros in `level.gc:2820-2826` (`play-desert`, `play-was`) are REPL convenience macros that call the player-lifecycle `start`, not the control-spec state. They're unreferenced by automated flow.

**control-spec is orthogonal to boot. Ignore.**

## 7. Does `target2.gc:379` and `(go target-title #f)` matter?

The brief cites `goal_src/jakx/engine/target/target2.gc:379` — but **jakx has no `target2.gc`** (not in decomp output, not in source tree). The `(go target-title #f)` line at `goal_src/jak3/engine/target/target2.gc:379` is inside jak3's `target-grab` state's `:event` handler, case `('change-mode) ('title)`. In jak3, when something sends `(send-event *target* 'change-mode 'title)`, the target goes to `target-title`, which in turn (per `levels/title/title-obs.gc:1106` `:enter`) spawns title-control.

**In our jakx flow we have bypassed this entirely.** `play` directly calls `(spawn-title-control)` before `initialize! 'boot`. The title-menu's `initialize! 'game` re-spawns the lambda, which calls `start`, which will (post-port) spawn target → target lands in `target-continue` by default. Nothing sends `'change-mode 'title` from title-menu. So the `target-title` path is only relevant if we later go for parity with the jak3 boot chain (Phase 3+ of the roadmap — swap `(spawn-title-control)` for the real target-title-driven spawn).

**For the minimum port: we do NOT need `target-title` state. target-continue is enough.** We can port `target-title` later as a 2-line stub state on target in the same file as target-continue.

## 8. Other risks / unknowns

- **Target type layout mismatch**: jakx `target-h.gc` is verbatim from jak3, but jakx's actual runtime size is `0x350` which is the same byte count but different method count (jakx target has ~60 methods vs jak3's ~70). If target-h's `(:methods ...)` declaration in our copy lists methods that don't exist in jakx's type-list, we get wrong method numbering. **Mitigation**: trim `:methods` to just methods 28 and 59 (explicitly named), drop the rest. Verify offset for method 59 survives.
- **Symbol-only types**: `fact-info-target`, `carry-info`, `water-control`, `joint-mod`, `joint-control`, `skeleton-group`, `collide-shape-moving` — if the deftype declares fields of these types but their deftypes aren't loaded in GAME.CGO before target-h loads, the link fails. We already hit this pattern in the title-control port. Check `goal_src/jakx/dgos/game.gd` for ordering.
- **`*setting-control*` field 3400** (`cam-default mode-name`): jakx's setting-control has this array; our porting work already covered it. Low risk.
- **Continue-point trans offset +12**: jakx uses `(+ s5 12)` for continue-point→trans. jak3 `continue-point` has `trans` at offset 0. If jakx reshuffled, the call breaks. Verify via decomp `continue-point-h`.

## 9. Shortest path summary

For Phase 3 "target spawns but doesn't drive gameplay":

1. Port `start` (real) — 15 lines. **A-level infra is enough.**
2. Port `target-init`, `reset-target-state`, `target-exit`, `target-no-move-post`, `target-post`, `target-real-post`, `level-setup`, `target-move-dist` — ~150 lines mechanical.
3. Port `target-init!` method — ~120 lines. **Medium risk**.
4. Stub `target-generic-event-handler` → `(lambda () 0)`. 2 lines.
5. Stub `target-continue` state → minimal (:code sleep-code :post target-no-move-post). 5 lines.
6. Trim target-h.gc externs and (:states ...) / (:methods ...) to match jakx reality.
7. Verify `*target-dead-pool*` + `*target-pool*` exist; create if not.

After that: `start` spawns target, target-init runs, target-init! sets up 2 collision primitives, reset-target-state zeros transform, target goes to target-continue, sleeps. `*target*` is non-#f. Title-menu's second `initialize! 'game` call goes through the 'play branch, re-spawns the lambda, lambda calls `start` again, which kills old target and spawns a fresh one. Process tree stays stable. No race yet, no visible player model, but the scaffolding is live.
