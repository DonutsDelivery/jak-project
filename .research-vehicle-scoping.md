# Minimal Vehicle Subsystem Port Scoping — Jak X Bringup

Research-only. Scope the shortest path from the in-flight target port to a single *vehicle* process existing at runtime. Not drivable, not visible, not physics-correct — just a live `wvehicle` (the concrete player-ridable type) that the target's `target-pilot-post` can copy a transform from.

## 0. How target meets vehicle (the key insight)

The relationship is **event-driven, not structural**. Target spawns first with no vehicle. Gameplay sends target a `(change-mode 'pilot vehicle-type pos quat)` event. `target-generic-event-handler` (jakx `target-handler_ir2.asm:328-495`) responds by:

1. Allocating a `vehicle-spawn-params` on stack (size 0x68 = 104 bytes).
2. Populating fields 0 (behavior = 4 = player), 88 (flags = 3), 93 (level-slot), 95 (driver-slot=12), view-index into field 92, position @ +16, rotation @ +32, velocity @ +48.
3. Calling `(vehicle-spawn vehicle-type *params*)` — the vehicle-manager global function, NOT a method.
4. On success, saving the vehicle handle into `*target*` field 184 (per `target-pilot-post` at `target-pilot_ir2.asm:82`).
5. `(enter-state target-pilot-start self)` — transitions target into the pilot-riding state defined in `pilot-states.gc`.

`target-pilot-start` :code is a `sleep` loop. Its `:post` (L1 in `pilot-states.gc`) calls `target-pilot-post`, which:
- Loads vehicle handle from target field 184.
- Copies vehicle position (root+12 = trans) to target root+12.
- Copies vehicle scale (root+60) to target root+60.
- Calls `quaternion-copy!` on vehicle root+28 → target root+28.

So from target's perspective, the vehicle owns the transform. Target just follows it. This is orthogonal to target-continue (the post-start idle state) and DOES NOT auto-trigger — something external must send `change-mode 'pilot`.

**For a minimum spawn, we need to either (a) send that event manually from REPL after target lands in target-continue, or (b) shortcut target-continue → spawn a bare vehicle-spawn-params → call vehicle-spawn directly.** Option (b) is cleaner for a smoke test.

### Target ↔ pilot ↔ vehicle relationship

- `target` holds field 184 = root (collide-shape-moving for its own input-only collision) — unchanged by pilot mode.
- When piloting, the *vehicle's* root (also a collide-shape-moving) provides the actual position/rotation that gets copied back to target each frame via `target-pilot-post`.
- There is NO `pilot-info` structure wired up at runtime. jakx `target-h.gc:93` declares `pilot-info basic` but no jakx file ever allocates it. The "pilot" connection is literally just: (a) target's field 252 = "pilot" symbol (driver-type marker), (b) the event `(change-mode 'pilot)` triggers spawning, (c) `target-pilot-post` reads from target field 184 which apparently gets reused.

**Correction to my Section 0**: target field 184 is `root` (collide-shape-moving). `target-pilot-post` reads `(+ s6 184)` = target root, then derefs `(+ v1 184)` where v1 is a process from a handle stored at target+252. That means target-pilot-post copies **from another process's root into self's root**. The handle source is target+252 (`(l.d (+ s6 252))`), which is set by target-generic-event-handler at line 389 (`sw v1, 252(s6)` — TODO re-verify: the MIPS shows `sd a0, 252(s6)` in pilot-states.gc anon-function 1 which is the `:enter` of target-pilot-start). So: target+252 = vehicle handle (8 bytes), written on enter-state target-pilot-start.

### racer-info

`racer-state` (race-h.gc:168) has `(racer handle)` at offset 16 pointing to the wvehicle process. `race-state` holds a 10-entry inline array `racer-array` + pointers to race-manager and hud. Racer-info is populated by `race-state-method-init-racers!` (race-h.gc:226, body in race-control). **None of this is required for a single vehicle spawn** — racer-info is the race coordinator layer ABOVE vehicle.

## 3. Entry point — what calls `vehicle-spawn`?

Three callers across the codebase:

1. **`target-handler_ir2.asm:437`** (`target-generic-event-handler`, case `'change-mode 'pilot`). This is the PLAYER path. Triggered externally.
2. **`net-player_ir2.asm:5018`** — multiplayer: when a remote player joins, their vehicle is locally spawned from a net-spawn packet. Skip.
3. **`net-deathrace_ir2.asm:6047, 6116`** — deathrace-mode variants. Skip.

**The single-player race flow** is: race-manager `setup-race` → for each racer, send the racer's process `change-mode 'pilot` OR directly call vehicle-spawn. The race-manager itself spawns the vehicle-manager via `vehicle-manager-start` if not present.

**Shortest path from live target:**
```
target exists (*target* non-#f, target-continue state)
  ↓ [REPL call or scripted]
send-event *target* 'change-mode 'pilot <vehicle-type> <position> <quaternion>
  ↓ target-generic-event-handler constructs vehicle-spawn-params on stack
  ↓ calls (vehicle-spawn <type> <params>)
vehicle-spawn (vehicle-manager.gc:vehicle-spawn)
  ↓ validates net-world, position magnitude, rotation norm
  ↓ if *vehicle-manager* is #f → (vehicle-manager-start *entity-pool*) creates it
  ↓ type-from-vehicle-type → resolves wcar-<type> class
  ↓ get-process from *vehicle-manager*'s heap
  ↓ run-function-in-process <proc> vehicle-init-by-other <type> <params>
  ↓ vehicle-init-by-other (method 131 of vehicle, ~500 lines) populates the allocated wvehicle:
      ├── looks up *car-level-array* / *car-info-default* → sets car-info pointer
      ├── calls (method 52 *entity-pool*) for extra-info allocation
      ├── sets rigid-body flags bit 27 (enable-physics)
      ├── calls (method 276 self) — likely init-collision!
      ├── copies params→position/rotation/velocity into root+12/+28/+60
      ├── sets vehicle-flags (+ self 724) from params behavior
      ├── sets control-mask (+ self 476) from params flags
      └── conditionally calls (method 724 self) = first state-method (idle/active)
  ↓ returns the new wvehicle process
target-handler stores vehicle handle in target+252, enters target-pilot-start state
```

## 5. Top 5 Blockers

Prioritized by impact × discovery difficulty. "jak3 parallel" = whether jak3 has a port we can crib.

1. **`wvehicle` deftype resolution** (Cat B/C). The type has 4660-byte layout with inline race-control (24 bytes at 2168), inline ai-state (136 bytes at 2192), inline wheel-state × 4 (288 bytes at 2480), inline engine-mix-state × 2 (64 bytes at 3792), inline jmod-steering-wheel (32 bytes at 4112), etc. All satellite types must be defined FIRST. Many overlap with the wvehicle-weapons* family. **jak3 parallel**: NO. jak3's vehicle is completely different. 100% jakx-unique port.
2. **`vehicle-init-by-other`** (Cat C). 515-line function that sets up ~40 fields on a freshly-allocated wvehicle, calls `(method 276 self)` = collision init, `(method 724 self)` = first state enter. Many field offsets (64, 184, 284, 440, 476, 724, etc.) must be verified against our deftype port. **jak3 parallel**: partial — jak3 has `vehicle-init-by-other` for traffic vehicles but it's jak3-shape. ~30% symbol overlap.
3. **`rigid-body-object` method bodies in `rigid-body-object.gc`** (Cat C, 3376 MIPS lines). The 27 methods (new, init-collision!, alloc-rbody-control!, init-rbody-control!, rbody-post, apply-momentum!, apply-damage, impulse-force<-penetrate, on-impact, impulse-handler, attack-handler, touch-handler, go-idle, go-active, etc.) are the spine of any physical-object process. Without at least `new`, `init-collision!`, `init-rbody-control!`, `rbody-post`, the vehicle can't be allocated or ticked. **jak3 parallel**: YES — jak3 `engine/physics/rigid-body.gc` has 90% overlap. High-value borrow.
4. **`vehicle-spawn` + `vehicle-manager-start`** (Cat C). `vehicle-spawn` is 622 lines with heavy net validation, debug assertions, process pool lookup, respawn-invul/driver/helmet sub-spawn, and the run-function-in-process handoff. `vehicle-manager-start` (124 lines) instantiates the singleton with a 28672-byte heap. **jak3 parallel**: partial — jak3 has `vehicle-spawn` but for traffic vehicles only, calling convention differs (jak3 takes traffic-object-spawn-params directly).
5. **`car-info` lookup table + `*car-level-array*` + `type-from-vehicle-type`** (Cat B). vehicle-init-by-other needs a car-info pointer picked from `*car-level-array*` or `*car-info-default*` based on spawn-params level-slot. Without this, the car-info field stays zero and any access via method 9..21 crashes. `type-from-vehicle-type` maps an enum to a concrete wcar-<type>. **jak3 parallel**: NO — jak3's cars are traffic background objects, jakx's are 15 hand-tuned racers with per-track variant. Must be data-driven from jakx files.

**Honorable mention blocker**: `wvehicle` has 32 slots for joint-mods (jmod-steering-wheel, jmod-frame × 2, jmod-axles × 4, etc.). These are inline `joint-mod-rotate-local` structs. If their deftypes aren't linked before wvehicle-h loads, compile fails.

10. **target-handler 'change-mode 'pilot branch verification** — check whether our future target-handler port (also an in-flight task for target subsystem) correctly implements the pilot branch. Low effort once target-handler lands.
11. **pilot-states + target-pilot port** (~200 lines total across both files — small). Defines `target-pilot-start` state and `target-pilot-post` post-hook. Critical glue but TINY. One-session.
12. **Build a REPL-callable `(test-spawn-vehicle)` harness** that manually constructs vehicle-spawn-params on the kernel heap, picks a default position, calls vehicle-spawn. Independent of target work; tests the vehicle pipeline in isolation.

## 7. Risks / unknowns

- **Method count drift**: jakx `vehicle` has 204 methods (29 inherited from rigid-body-object counting methods 30-58 + 115 own methods 89-203). Any method renaming between types changes the table; we have to keep vtable offsets exact.
- **Size asserts**: wvehicle :size-assert #x1234. Any field mismatch blows it. 272 methods × many inline satellites means ONE missed field offset cascades into hundreds of broken references.
- **run-function-in-process mechanics**: vehicle-spawn allocates via `get-process *vehicle-manager*`, then invokes `(run-function-in-process proc vehicle-init-by-other type params)` — our jakx process infra needs to support this pattern. Jak3 version is identical; should be already in OpenGOAL kernel.
- **`*vehicle-manager*` heap**: 28672 bytes is hardcoded. If we're memory-tight at boot, vehicle-manager-start may fail silently — need to verify kernel has that much free heap when called.
- **Vehicle-manager as singleton vs. race-manager dependency**: vehicle-manager is spawned lazily by vehicle-spawn (jakx pattern). race-manager is separate. Good — no forced ordering.
- **Net code entanglement**: vehicle-spawn has many `*net-mgr*` / `*net-world*` / `*net-players*` branches. Stubbing these to #f bypasses the network path, but `sb v1, 92(s4)` (player-id from view index) must still succeed. The sllv+daddu at lines 91-92 expects net-players to be a real array — forcing #f there goes through the fallback branch cleanly.
- **Level slot lookup**: vehicle-init-by-other picks level from `(+ *level* (+ (* 1440 params->level-slot) 544))` — so we need the level-manager's level-array layout intact, with slot -1 as a valid "don't set level" marker.

## Cross-cutting concerns with the target port

- **Shared `collide-shape-moving` infra.** Target research (`.research-target-minimal-port.md` §5) already flagged this as the target-side dependency. The same code unlocks vehicle, since wvehicle is ALSO a process-focusable allocating a collide-shape-moving via `new 'process`. Whichever port lands first unblocks the other.

- **Kernel heap pressure.** Target adds ~848 bytes. A single wvehicle adds ~4660 bytes + 28672 for vehicle-manager = ~34KB. Combined with existing title-control allocations, may hit kernel-dram-stack ceiling earlier than expected. Profile during session 3.
