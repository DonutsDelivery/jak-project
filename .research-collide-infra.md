## Notes

**1, 2 — collide-shape-moving / prim-group / prim-sphere:** All deftypes plus their `new` methods exist in jakx with full bodies. Inheritance chain is intact (`collide-shape-prim` base `new` also present at `collide-shape-h.gc:429`). Constructor cascade from `target-init!` (prim-group → N prim-sphere children) will compile.

**3 — method 132:** Re-check the premise. Jak3's `collide-shape-moving` has `:method-count-assert 68` (max 67); the jakx commented deftype in `decompiler/config/jakx/all-types.gc:30023` also shows max 66. No method 132 exists on this class. Checking jak3's `target-init!` body (`goal_src/jak3/engine/target/logic-target.gc:2970`) shows it calls `move-to-point!` — this is method 49 (inherited from `collide-shape`), not 132. Recommend the caller re-identify the method they intended.

**`move-to-point!` caveat (important for the port):** Declared on `collide-shape` in jakx at `collide-shape-h.gc:349`, but the `defmethod` body is **NOT present** in `goal_src/jakx/engine/collide/`. Callsites in jakx exist (`generic-obs.gc:137,203,214,697,2509,2557`, `projectile.gc:559`) — they will currently resolve via whatever runtime state the method slot is in. Body is in `goal_src/jak3/engine/collide/collide-shape.gc:2158`. If `target-init!` ends with `(move-to-point! (-> this control) (-> arg0 trans))` like jak3's, this is **B** — extern/declared but no real body; port of `move-to-point!` itself likely needed too. Same applies to `move-by-vector!`, `debug-draw`, `integrate-and-collide!`, `build-bounding-box-for-shape`, etc. — the whole `collide-shape.gc` source file is effectively absent from jakx (the placeholder `collide.gc` is 32 bytes).

**4 — cshape-reaction-default:** jakx uses the symbol but never defines the function. jak3 body at `collide-shape.gc:990` is ~80 lines, depends on `cshape-reaction-update-state` (also not in jakx). Since `target-init!` assigns `target-collision-reaction` (not the default) to `(-> s0-0 reaction)`, it isn't a blocker for the minimal port — but it is a blocker for the touching generic-obs callsites that reference it.

**5 — *standard-dynamics*:** Real static instance, not an extern. Fields `gravity`, `gravity-normal` used at 7 jakx callsites. Ready.

**6 — view-get-active-index:** Declared as extern in jakx all-types (`(function int)`). No source body in any game. It is a jakx-specific function (the jak3/jak2 equivalent uses different camera APIs). Since `target-init!` itself doesn't call it directly, not a blocker for the minimal port, BUT if porting the `cam-default mode-name` plumbing from jak3's `target-init!`, note that jak3 line 2861 uses `(set! (-> *setting-control* cam-default mode-name) 'cam-string)` — that path doesn't need `view-get-active-index`. If the user's plan specifically requires storing active-view index, this needs a manual design decision.

**7 — buzz-stop!:** Defined in `pad.gc`. jak3's `target-init!` calls it at line 2827 (right before the method definition).

**8 — set-continue!:** Full implementation present. Referenced throughout jakx code (level.gc, scene.gc, script.gc, game-save.gc, game-info.gc).

## Summary for Port Readiness

Green-light for the `new 'process 'collide-shape-moving` / prim-group / prim-sphere allocation chain. `*standard-dynamics*`, `buzz-stop!`, `set-continue!` all good. The two real blockers for a faithful port of jak3-style `target-init!`:
1. `move-to-point!` has no jakx body (B — port from jak3 `collide-shape.gc:2158`).
2. `cshape-reaction-default` has no jakx body (B — port from jak3 `collide-shape.gc:990`), though avoidable if `target-init!` uses a custom reaction function.

`view-get-active-index` is the only D (missing, no source reference anywhere) — manual check needed only if the port actually invokes it.
