# OpenGOAL Porting Research — Applied to Jak X

Scope: how the OpenGOAL team got Jak 1, 2, and 3 running, and what that tells us about the shortest path from our current Jak X state (empty display-loop rendering colored quads) to a visible title screen.

Most authoritative sources found: the local repo (`docs/`, `goal_src/jak3/`), the upstream porting docs at `opengoal.dev/docs/porting-info/*`, and the Mega Issue #1570 for Jak 2 bringup. OpenGOAL does **not** publish a step-by-step "boot to title" recipe — the information below is reconstructed from those sources plus the canonical call path in `goal_src/jak3/`.

---

## Section 1: OpenGOAL porting history

### Jak 1 (~2020 – 2022, shipped)
First game. The team built everything from scratch in parallel — language, compiler, decompiler, runtime. Current status: "polished, complete" (per README). `docs/progress-notes/jak1/changelog.md` documents the compiler from v0.1 through v1.0+; `docs/progress-notes/jak1/code_status.md` shows the kernel/engine-early decomp status (gcommon → gkernel → math → vector/matrix/quaternion → dma/dma-buffer/dma-bucket → pad → gs → display-h → video-h). That ordering — roughly *common types → kernel process tree → math → DMA infrastructure → display* — is the implicit canonical bringup order, and it's reflected in `goal_src/jak3/kernel/` and the engine directory layout.

### Jak 2 (2022 – Nov 2023 public beta, ~18 months)
Second game. Issue #1570 ("[jak2] Mega Issue for setting up decomp") captures the launch plan, which is the clearest explicit description of bringup I found. Key moves:
1. Decompiler + compiler upgrades (stack vars, multi-project support, all-types generation)
2. Multi-game C++ runtime: "have a jak 1 and a jak 2 build at the same time, with as much shared c++ code as possible"
3. "Working goalc for jak2" + skeleton `goal_src/` directory
4. Kernel decomp + tests — verify core functionality runs
5. Engine bringup; graphics sync / sound / OVERLORD deferred behind gameplay code

Public beta: Nov 2, 2023 (per PCGamingWiki).

### Jak 3 (Oct 2023 – 2025 beta, ~18–24 months)
Third game. Oct 2023 progress report: "Allow Decompiler To Run on Jak 3 Code and Improve `all-types` Generation" plus v5 data-file format support. January 2024: "finished over 100 of the early engine files in under 10 days" because they were largely identical to Jak 2 — but *"It will be a while before the game can actually boot."* Q2 2025: "most missions are playable." Q1 2026: beta, v0.3.1 release.

### Pattern across all three
The team **does not** try to get the game booting as an early milestone. The order is reliably:
1. Decompiler + compiler can process the game
2. C kernel and all-types generation work
3. Bulk engine-file decomp (shared with prior game) — hundreds of files before first boot
4. Graphics and sound bringup happen *late*, after engine code is present
5. First boot, then title, then gameplay come at the end

PR #3965 ("jakx: Kernel work") is the upstream analogue of our own work — a draft bringing the Jak X C-kernel in line with the Jak 3 baseline. Per the fetched description, it copied Jak 3 code as a buildable foundation then incrementally layered Jak-X specifics on top. Still draft; UNDERLRD.IRX and some kmachine functions remain.

## Section 2: Canonical bringup order (reconstructed from jak3 source)

The actual "boot to title" call path in Jak 3 is:

1. **C kernel boot** (`game/kernel/jakx/kboot.cpp` et al.) — allocates GOAL heaps, loads KERNEL.CGO, calls `kernel-dispatcher`.
2. **`kernel-dispatcher`** (`goal_src/jak3/kernel/gkernel.gc:1516`) — runs the listener function if any, then `execute-process-tree` on `*active-pool*`.
3. **`play-boot`** (`goal_src/jak3/engine/level/level.gc:2914`) — called by the C kernel when no listener command is provided. It spawns a process that calls `(play #t #t)`.
4. **`play`** (`engine/level/level.gc:2828`) — "Start (or restart) the game! This will start up the display process, and load the initial level." Key ordering inside `play`:
   - `stop 'play`
   - `init-level-system *level* #t`
   - picks default boot level: `title` in release, `*default-boot-level*` with debug segment
   - loads the title level via `level-get-for-use` + `load-continue` loop
   - `(on #t)` — turns the display process on
   - `initialize! *game-info* 'boot …` — **this is the boot-mode path through `game-info-method`, which is exactly what our task has commented out**
5. **`target-title`** state (`engine/target/target-death.gc`, `target2.gc`) — triggered by `go target-title #f`. Its `:enter` calls `(initialize! *game-info* 'game …)`, kills any existing `title-control`, and spawns a new one via `process-spawn title-control :init title-control-init :name "title-control" :to *entity-pool*` (title-obs.gc:1151).
6. **`title-control-init`** (`levels/title/title-obs.gc:994`) — gets the `title` level, applies settings (`allow-pause`, `airlock`, `music`, `entity-name "camera-311"`), sets up two `external-art-buffer`s, initializes sprites, and finally `(go-virtual startup)`.
7. **`title-control` `startup` state** — draws the logo and handles the "press start" input.

The subsystems that MUST be alive for this chain to work:

| Subsystem | Needed for |
|---|---|
| Process tree / kernel-dispatcher | step 2 |
| `*active-pool*`, `*4k-dead-pool*`, `*target-dead-pool*`, `*entity-pool*` | steps 3, 5, 6 |
| `*display*` + display-process (`(on #t)`, `frame-counter`, clocks) | step 4 and everything that reads `*display* base-clock` (i.e. everything) |
| `*level*` system + `load-state` + level DGO loading | step 4 (load title level) |
| `*setting-control*` with a real `speech-control`, and its `update` method | step 4 (`apply-settings`) and every subsequent `set-setting!` |
| `*game-info*` with `initialize!` in `'boot` mode working all the way through | step 4 tail |
| `*time-of-day*`, `*mood-control*`, `*load-state*` | inside `play` |
| Texture pool / DirectRenderer for fonts + sprites | step 7 (logo + "press start" text) |
| `*title-control*` process + `external-art-buffer` allocation | step 6 |

Note the display-process + `*setting-control* update` is the hot loop. It runs every frame regardless of whether a title is spawned — the engine is *supposed to* render "nothing" for several frames before the title spawns and loads. So "empty buckets / black screen" is the correct state until `play-boot` completes.

## Section 3: Recommendations for Jak X — next 3 – 5 concrete steps

Given the known state (1261/3050 deftypes live, display-loop runs, can draw colored quads, `*pc-settings*` real, `setting-control update` crashes on null `speech-control`, `game-info initialize! 'boot` commented out, no `title-control` spawn), the highest-ratio moves — in order — are:

### 1. Stub `*speech-control*` as a real (empty-method) object so `setting-control update` stops crashing
You already have `*setting-control* update` running every frame but calling `speech-control-method-11 *speech-control*` on a null pointer (line 1588 of jak3's `settings.gc`). Fix: `(define *speech-control* (new 'global 'speech-control))` with a trivial `speech-control-method-11` that returns `#f`. Per MEMORY.md's "proper fixes over commenting out" note: add real type stubs with the required field layout; do not comment out the call site. **Unblocks:** the real settings pipeline (volumes, music, language, borders) — every later system queries `*setting-control* user-current`.

### 2. Uncomment `game-info initialize!` boot-mode path and stub the missing dependencies one at a time
`initialize! *game-info* 'boot` (jak3 game-info.gc:404) touches `*FACT-bank*`, `*GAME-bank*`, `*bigmap*`, `sub-task-list`, `continue-point`, task masks. Strategy copied from Jak 3 bringup (Q1 2024 report: *"finished over 100 of the early engine files in under 10 days"*): don't decomp these all properly — stub each referenced global as a `new 'global` with zero-initialized fields and empty-list defaults, then uncomment `initialize!`, let it crash, stub the next missing thing, repeat. **Unblocks:** everything downstream of "game-info is real," which includes target spawn, HUD, controller input routing.

### 3. Prove the texture pipeline with a single font glyph before anything else
"Font shows noise" means texture upload is running but data is wrong. Before writing `title-control`, isolate this: pick one ASCII glyph from `common` texture pages, dump the raw GS-format bytes, trace through `TexturePool` upload into OpenGL, and confirm the decoded RGBA matches the PS2 expected. The jak3 `opengoal.dev/docs/porting-info/graphics/` page is clear that DirectRenderer handles "debug, fonts, UI" — the same code path the title uses. Fix this before stage 4. **Unblocks:** title logo, "press start" text, debug HUD overlays, all menus.

### 4. Spawn `title-control` manually from the REPL before wiring `play-boot`
Don't try to get `play-boot → play → target-title → title-control` all wired at once. From the REPL: `(process-spawn title-control :init title-control-init :name "title-control" :to *entity-pool*)` and see what explodes. You'll get a focused, small list of missing symbols (`external-art-buffer`, `*game-info* dust-storm`, `hud-sprite-flags`, `*font-default-matrix*`, `sprites 0 color-ptr`). Stub them. This is exactly the Jak-2 pattern of "working goalc + minimum engine skeleton" from issue #1570 step 3. **Unblocks:** visible title screen without needing boot chain correctness.

### 5. Only *after* step 4 shows pixels: wire `play-boot` properly
Once a manually-spawned `title-control` renders, re-enable `play-boot` → `play` → `target-title` → auto-spawn. Do NOT do this before; the chain will fail silently and you'll lose signal on which link is broken.

**Explicit non-priorities** (defer, per the Jak 2 bringup pattern): graphics synchronization polish, OVERLORD streaming, sound RPCs (`PLAY`, `STR`), real level-load DGOs beyond `title`. The Oct 2023 Jak 3 report and the `docs/porting-info/porting_to_x86` page both list these as explicitly deferred behind engine bringup.

## Section 4: Tools and references

### Tools / scripts already in-repo that we may be underusing
- **`collect_errors.sh`** (repo root) — aggregates decompiler failures; useful for tracking the "305 files failing" count.
- **`restore_engine_files.py`** (repo root) — exists precisely for the "progressively enable more deftypes/engine files" workflow.
- **`task repl`** / `scripts/shell/gc.sh` — REPL that attaches to the running game. Critical for step 4 above: spawn `title-control` interactively instead of via boot chain.
- **`scripts/shell/decomp.sh`** — re-run decompiler after updating all-types. The Jak 2/3 pattern was to edit `all-types.gc` → re-decomp → compile → test. Our 305 failing files likely shrink fast with all-types updates.
- **`-nokernel`** runtime flag (changelog v0.1) — lets the runtime run without KERNEL.CGO. Useful for debugging before the full kernel boots. Also `(-nodisplay` etc. — see `kboot` options).
- **Debug menu / DirectRenderer** — per the `opengl_renderer/debug_gui.cpp` hit, you can overlay debug text before any GOAL text rendering works. Put frame counter + proc tree dump on screen for "something is running" signal.
- **REPL `get-info`, `autocomplete`, `:disassemble`** (changelog v0.6, v0.2) — use these to probe state of the running game rather than restarting.
- **Debugger** (`opengoal.dev/docs/developing/debugging/`) — OpenGOAL has its own debugger invoked from the compiler prompt; likely more useful than gdb for GOAL-side crashes.

### Not found / unknowns
- There is **no upstream "boot to title screen checklist"** document. I searched opengoal.dev, the repo `docs/`, and the jak2/jak3 progress notes; none publish an ordered list.
- No blog post calls out the exact PR where Jak 2 or Jak 3 first showed a title screen. Git log may contain it but requires walking commits — the closest public signal is the Nov 2023 Jak 2 public-beta announcement.
- Upstream has not committed to decompiling Jak X; per the Q1 2026 FAQ, "it's unsure whether Jak X will be decompiled." This port is essentially on its own for anything Jak-X-specific (racing, combat, HUD, IOP UNDERLRD).
